# Internationalization (i18n) Guide

Both demo chatbots (`ai-chatbot` and `api-chatbot`) load all user-facing
strings from JSON translation files. The active language is selected at
container startup with the `APP_LANG` environment variable
(default: `en`). `APP_LANG` is used instead of the standard POSIX `LANG`
variable to avoid colliding with the system locale.

## Layout

```
kubernetes/
‚îú‚îÄ‚îÄ app/i18n/                    # ai-chatbot (Network Intercept) translations
‚îÇ   ‚îú‚îÄ‚îÄ en.json                  # ‚Üê default, also the canonical key list
‚îÇ   ‚îî‚îÄ‚îÄ pl.json
‚îî‚îÄ‚îÄ api-chatbot/i18n/            # api-chatbot (API Runtime) translations
    ‚îú‚îÄ‚îÄ en.json
    ‚îî‚îÄ‚îÄ pl.json
```

Translations live **inside the container image** (copied in at build
time). Switching language at runtime is therefore a `kubectl set env` +
rolling restart ‚Äì no rebuild required, as long as the target translation
file is already baked into the image.

## Switching language on a running cluster

```bash
./scripts/switch-language.sh        # show current state + available languages
./scripts/switch-language.sh en     # English (default)
./scripts/switch-language.sh pl     # Polish
./scripts/switch-language.sh de     # any code with a matching <code>.json
```

The script:

1. Sets `APP_LANG=<code>` on both deployments
2. Rolls a restart of both
3. Waits for the new pods to become ready
4. Prints a final summary

If a translation file is missing inside the image, the application falls
back to `en.json` at startup and logs a warning.

## Adding a new language

1. Copy the English files as your starting template:

   ```bash
   cp kubernetes/app/i18n/en.json         kubernetes/app/i18n/<code>.json
   cp kubernetes/api-chatbot/i18n/en.json kubernetes/api-chatbot/i18n/<code>.json
   ```

2. Translate **only the values**. Keep every key (including `_meta`)
   exactly as-is ‚Äì the application looks up strings by key.

3. Update the `_meta` block in your new file:

   ```json
   "_meta": {
     "lang_code": "<code>",
     "html_lang": "<code>",      // BCP-47 tag, e.g. "de" or "fr-CA"
     "js_locale": "<locale>"     // for new Date().toLocaleTimeString(...)
   }
   ```

4. Rebuild and push the images so the new file lives in the next pod:

   ```bash
   ./scripts/deploy-app.sh
   ```

5. Switch:

   ```bash
   ./scripts/switch-language.sh <code>
   ```

## Key reference

The English files (`en.json`) are the canonical source of truth for the
key set. If you rename or add a key in one language, mirror that change
in every other language file (or new keys will simply render as
`undefined` / blank).

| Section          | Used in                                                 |
| ---------------- | ------------------------------------------------------- |
| `_meta`          | `<html lang>`, JS `toLocaleTimeString` locale           |
| `page.*`         | `<title>`, header, badges                               |
| `flow.*`         | Traffic-flow diagram steps                              |
| `welcome.*`      | First-message panel                                     |
| `input.*`        | Textarea placeholder, send button, attack button title  |
| `client.*`       | Browser-side JS messages (errors, demo labels, tags)    |
| `system.*`       | Server-side errors, log lines, default system prompt    |
| `attack_prompts` | (ai-chatbot only) malicious sample prompts for the demo |
| `status_bar.*`   | (api-chatbot only) top status bar                       |
| `side_panel.*`   | (api-chatbot only) right-hand panel headings            |

## Notes

- **System prompt for Gemini (ai-chatbot)**: `SYSTEM_PROMPT` env var
  takes precedence. If unset, the prompt is taken from
  `system.default_system_prompt` of the active language.
- **AIRS block messages (api-chatbot)**: `system.blocked_prompt_message`
  and `system.blocked_response_message` are returned to the browser
  whenever AIRS blocks content ‚Äì translate these so end-users see the
  block reason in the chosen language.
- **Logs**: log strings are also localized. Operators reading container
  logs may want to keep `APP_LANG=en` even if the UI runs in another
  language (in that case, deploy two replicas with different `APP_LANG`
  ‚Äì or just translate the `system.log_*` keys carefully).
